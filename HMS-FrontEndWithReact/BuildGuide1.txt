COMPREHENSIVE MASTER INSTRUCTION DOCUMENT
Patient-Only React Frontend for Multi-Clinic Hospital Management System (HMS)

SECTION 0: ABSOLUTE CONTEXT & CRITICAL UNDERSTANDING
0.1 System Architecture Overview
You are designing and implementing a pure React frontend UI that serves EXCLUSIVELY PATIENT USERS of a sophisticated multi-clinic Hospital Management System (HMS).
Critical Facts You Must Internalize:

Backend Reality: A fully functional Laravel backend exists with a complete relational database schema (MySQL/MariaDB)
Separation of Concerns: All administrative, clinical staff (doctors, nurses, lab technicians, pharmacists, receptionists, accountants), and internal management dashboards are already built using Laravel Blade templates on a separate domain
This Frontend's Purpose: This React application is a patient-facing portal ONLY, deployed independently, and communicates with the Laravel backend exclusively through RESTful APIs
Your Boundaries: You must NOT redesign backend logic, invent new workflows, assume administrative permissions, replicate internal staff functionality, or expose write access to protected medical data

The Laravel Backend is the Single Source of Truth. You are building a view layer only.
0.2 Database Schema Understanding (Critical)
The backend database uses the prefix lara_ for all tables. Key tables you must understand:
Core Identity & Access:

lara_users - User accounts (includes patients, doctors, staff)
lara_patients - Patient-specific data and demographics
lara_clinics - Multiple clinic/branch locations
lara_roles / lara_permissions - Role-based access control

Clinical Operations:

lara_appointments - Appointment scheduling
lara_visits - Actual patient visits (linked to appointments)
lara_consultations - Doctor consultation records
lara_admissions - Inpatient admission records
lara_beds / lara_bed_assignments - Bed management
lara_prescriptions / lara_prescription_items - Medication prescriptions
lara_lab_test_orders / lara_lab_test_results - Laboratory tests
lara_patient_vitals - Vital signs recordings
lara_patient_medical_history - Historical medical conditions
lara_patient_allergies - Allergy records

Supporting Systems:

lara_departments - Hospital departments
lara_doctors - Doctor profiles and specializations
lara_medicines - Medicine catalog
lara_invoices / lara_invoice_items / lara_payments - Billing
lara_pharmacy_sales - Pharmacy transactions

0.3 Multi-Clinic Architecture (CRITICAL)
Understanding Clinic Context:

Multiple Clinic Reality: Patients can be registered at multiple clinic locations
Single Active Context: Patients access data from ONE clinic at a time
Clinic-Scoped Data: Nearly every table has a clinic_id foreign key
Data Isolation: Backend APIs MUST filter all queries by the active clinic context
Context Switching: Changing clinics requires clearing cached data and refetching

Schema Evidence:

lara_patients.clinic_id - Primary clinic association
lara_doctor_clinic - Many-to-many doctor-clinic relationships
lara_appointments.clinic_id - Clinic-scoped appointments
lara_admissions.clinic_id - Clinic-scoped admissions
All major transactional tables include clinic_id

0.4 Data Access Philosophy
Read-Only Medical Data:
Patients can VIEW but NOT EDIT:

Diagnoses
Prescriptions
Lab results
Doctor notes
Vital signs
Medical history

Limited Write Access:
Patients can potentially:

Book appointments
Cancel appointments (within constraints)
Update personal contact information
Change password

Zero Access:

Doctor schedules (except availability)
Billing calculations
Staff operations
Administrative functions
Cross-patient data

2.7.2 Timing Functions
Standard Easing:

Ease Out (Deceleration): cubic-bezier(0.0, 0.0, 0.2, 1) - Use for entrances
Ease In (Acceleration): cubic-bezier(0.4, 0.0, 1, 1) - Use for exits
Ease In-Out (Smooth): cubic-bezier(0.4, 0.0, 0.2, 1) - Use for transitions

Medical-Specific:

Calm Ease: cubic-bezier(0.25, 0.1, 0.25, 1) - Gentle, non-aggressive
Linear: Only for loaders and continuous animations

2.7.3 Duration Standards
Micro Interactions (Hovers, Focus):

Duration: 150ms - 200ms
Easing: Ease Out

Component Transitions (Expand/Collapse, Show/Hide):

Duration: 250ms - 300ms
Easing: Ease In-Out

Page Transitions (Route Changes):

Duration: 300ms - 400ms
Easing: Ease Out

Loading States (Spinners, Progress):

Duration: Continuous or 500ms minimum
Easing: Linear or Ease In-Out

2.7.4 Allowed Animations

Progress Indicators: Loading spinners, progress bars
Status Changes: Color transitions for status badge updates

FORBIDDEN:

Bounce effects
Spring animations
Overshoot/elastic easing
Rotation (except loading spinners)
3D transforms
Parallax scrolling
Confetti/celebration animations
Shake/vibrate effects

3.3.2 State Management Rules
Context Switching Rules:
When user changes activeClinic:

Clear all cached clinic-specific data
Set global loading state
Force refetch of dashboard data
Reset all section-specific state
Clear error states
Close all open modals
Redirect to dashboard

Token Management:

Store token in memory (state) + httpOnly cookie (if possible)
Include token in all API requests via interceptor
Refresh token before expiration (if backend supports)
Clear token on logout or 401 response
Redirect to login on authentication failure

Error Handling:

API errors populate errors state
UI components read error state and display

Global error boundary catches React errors

4.1.4 Login Flow Logic
User Action Sequence:

User enters email/phone
User enters password
User clicks "Sign In"
Frontend validates input (client-side)
If valid:

Button shows loading state
API call: POST /api/patient/login
Request body: { email_or_phone, password }


On Success:

Receive: { token, user, patient }
Store token in authContext and secure storage
Fetch available clinics: GET /api/patient/clinics
Redirect to Clinic Selection page


On Failure:

Display error message
Clear password field
Reset button to normal state
Optionally: Rate limit attempts (3 attempts = temp lockout message)

SECTION 16: LOADING STATES & PERFORMANCE
16.1 Loading State Principles
Purpose:

Inform users that data is being fetched
Reduce perceived wait time
Maintain UI structure during load

Design Rules:

Never block entire UI with full-page spinners (except auth actions)
Use skeletons for content areas
Use spinners for button actions
Preserve layout (no shifting)


16.2 Skeleton Screens
When to Use:

Initial page load
Tab switching
Filter application
Pagination

Design:

Background: #E8EEF3
Animation: Shimmer effect (subtle gradient sweep)
Border Radius: Match actual content
Height: Approximate final content height

Skeleton Elements:

Text Line Skeleton:

Rectangle with rounded corners
Width: Variable (full, 80%, 60% for varied look)
Height: Based on text size


Card Skeleton:

Card-shaped rectangle
Internal lines representing text
Circular shapes for avatars/icons


Image Skeleton:

Square or rectangle with aspect ratio
Optional: Icon in center



Implementation:
javascript// Example pseudo-structure
<SkeletonCard>
  <SkeletonCircle size={48} /> // Avatar
  <SkeletonLine width="80%" /> // Title
  <SkeletonLine width="60%" /> // Subtitle
  <SkeletonLine width="40%" /> // Meta info
</SkeletonCard>

16.3 Loading Spinners
When to Use:

Button actions (login, save, submit)
Small data refreshes
Modal content loading

Design:

Circular Spinner:

Circle with rotating border segment
Size: 16px (small), 24px (medium), 32px (large)
Color: Match context (white on colored buttons, #2C5F8D on white)
Animation: Smooth 1s rotation, linear easing



Button Loading State:

Text changes: "Submit" → "Submitting..."
Spinner appears left of text
Button disabled during load
Width remains constant (no layout shift)


16.4 Progress Indicators
When to Use:

Multi-step forms
File uploads
Long-running operations

Design:

Progress Bar:

Height: 4px
Background: #E8EEF3
Fill: #2C5F8D
Border Radius: 2px
Animated fill (transition 300ms)



Percentage Display:

Text: "Uploading... 45%"
Position: Above or beside progress bar


16.5 Optimistic UI (Use Sparingly)
When NOT to Use:

Medical data (appointments, prescriptions, lab results)
Financial transactions
Critical status changes

When OK to Use:

SECTION 18: SECURITY & DATA PRIVACY
18.1 Frontend Security Principles
Never Trust User Input:

All input validated client-side AND server-side
Client-side validation for UX only
Server enforces security

Never Expose Sensitive Data:

Don't include patient_id explicitly in API requests (derive from token)
Don't log sensitive data
Don't store unencrypted data in localStorage

HTTPS Only:

All API calls over HTTPS
No mixed content


18.2 Authentication Token Handling
Token Storage:

Option A (Recommended): httpOnly cookie (set by backend)
Option B: Memory (React state) + Refresh Token in httpOnly cookie
NOT Recommended: localStorage (XSS vulnerability)

Token Inclusion:

Send token in Authorization header
Axios interceptor to add token to all requests

javascriptaxios.interceptors.request.use(config => {
  const token = authContext.token;
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
Token Expiration:

Check token expiration before requests
Implement refresh token flow (if backend supports)
On 401 response: Clear auth state, redirect to login


18.3 Data Privacy Compliance
HIPAA/GDPR Considerations (if applicable):

Display privacy policy link
Allow users to request data export
Allow users to request account deletion (if backend supports)
Log user consent for data processing

Data Minimization:

Only request/display necessary data
Don't cache sensitive medical data indefinitely
Clear sensitive data on logout

Audit Logging:

Backend logs all access (frontend just makes requests)
Display "Last accessed" timestamps (if backend provides)


18.4 XSS Prevention
React's Built-in Protection:

React escapes JSX content by default
Avoid dangerouslySetInnerHTML unless absolutely necessary

If HTML Rendering Required:

Sanitize HTML with library (DOMPurify)

javascriptimport DOMPurify from 'dompurify';
const cleanHTML = DOMPurify.sanitize(dirtyHTML);
return <div dangerouslySetInnerHTML={{ __html: cleanHTML }} />;

18.5 CSRF Protection
Backend Responsibility:

Backend implements CSRF tokens
Frontend includes token in requests (if required)

SameSite Cookies:

Backend sets cookies with SameSite=Strict or Lax


18.6 Rate Limiting
Client-Side:

Debounce search inputs
Throttle button clicks
Prevent multiple simultaneous submissions

Backend Enforcement:

Backend rate limits API requests
Frontend displays rate limit errors gracefully

Vite/Webpack handles automatically in production
